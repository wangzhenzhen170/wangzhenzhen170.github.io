git项目中常用操作

上篇简单介绍了git的工作原理，这篇简单整理了我在实际项目开发中常用的操作

版本管理

git add 把文件添加到暂存区
git commit -m ‘’ 把暂存区的所有内容提交到当前分支
如果不add 到暂存区直接commit，那么就不会加入到commit中
git diff 查看工作区和版本区的区别
git checkout -文件名 让这个文件回到最近一次commit或者add 时的状态
版本回滚
git log 显示从最近到最远的提交日志
git log –pretty=online 提交日志一行显示
git reset –hard ‘版本id’ 指针commit信息和文件都回到指定的版本
git reset –soft ‘版本id’ 只有指针commit信息回到制定的版本，本地的文件并没有回到制定版本
在用git log已经找不到回到未来的版本号怎么办
git reflog 用来记录每一次命令，确定要回到未来的那个版本

分支常用命令

每次提交，git都把他们串成一条时间线，这条线就是我们创建版本库的时候默认的主分支master，HEAD好似一个游标，指向master，master才是指向提交的，HEAD指向的就是当前分支
每次提交，master分支都会向前移动一步，随之不断的提交，master分支线也越来越长
git checkout –b dev 创建并切换到dev分支
git push origin dev 把本地分支dev提交到远程分支
git branch 查看本地分支
git branch -a 查看本地和远程仓库所有分支
git branch -D dev 删除本地分支dev
git push origin –delete dev 删除远程分支
git checkout -b dev origin/dev 切换远程dev分支，在本地起名为dev分支，并切换到到本地dev分支上

分支合并

例如将dev分支合并到master分支上

  • git checkout master切换到master分支
  • 合并指定的dev分支合并到当前分支（master）上 merge请求有两种方式：
    1）git merge dev
    这是Fast-forward快速合并模式 也就是直接把master指向dev的当前提交，一般不用这用这种方式
    2）git merge –-no-ff –m ‘合并信息’ dev
    推荐这种，git就会在merge时生成一个新的commit，这样就可以从历史分支上看出分支信息
  • git pull
    拉取master最新代码，当master分支和dev分支上对同一个文件都有改动时合并那一定会出现冲突，需要我们手动解决冲突
  • git push
    如果有冲突，解决完冲突后提交到远程，那两个分支合并完成

项目中分支管理

在实际开发中，我们应该按照几个基本原则进行分支管理

  • 首先 主分支master应该是非常稳定的，也就是仅仅用来发布新版本，平时不能此分支工作
  • 我们在dev(名字自己取)分支工作，这个分支是不稳定的，发版的时候把dev分支合并到master，在master主分支发版
  • 项目中我们都是多人合作，每个人都在dev分支上工作么，NO！我们每个人在做新功能或修复bug的时候，都会在本地开一个新的分支，完成后在合并到dev分支上，这样的好处是，如果你新做的功能，产品突然改变了主意，此次不发版，或者这个新功能没做完有更紧急的功能需要做，你就可以很轻松的切换分支就可以搞定了

标签tag管理

git tag –a v1.0 –m ‘标签的说明’ 创建标签v1.0
git push v1.0 origin 将tag推送到远程分支
git tag 查看所有标签
git log –pretty=oneline –abbrev-commit 查看历史提交
git tag v1.0 ‘commitId’ 查看历史提交的commit id ，在此id打标签
git push origin v1.0 将标签v1.0推送到远程
git push origin –-tags 一次推送全部尚未推送到远程的本地分支
git tag –d -v0.1 删除本地标签v1.0
git push origin :refs/tags/ v1.0 删除远程分支（需先将本地分支删除）

版本回退

例如:远程dev分支上有两个v1.0和v2.0两个tag，当前是v2.0,但是我想将dev分支回退到v1.0，将2.0新添加的功能舍弃掉

  • git checkout -b dev2.0(分支名字自己随便取) 首先新开一个本地分支将v2.0备份
  • git checkout develop 切回develop分支
  • git show v1.0 查看v1.0的tag信息，找到commit id号
  • git reset –hard id (v1.0的id) 将文件和指针都回退到v1.0状态
  • git reset –soft id （v2.0的id）将指针回退到2.0版本，但文件还是还是处在1.0的状态
  • git add .
  • git commit -m ‘’
  • git push 推送到远程
    这样develop分支上的文件就回到了v1.0时的状态，分支dev2.0是V2.0的备份
